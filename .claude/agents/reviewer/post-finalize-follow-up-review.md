# Reviewer — post-finalize follow-up reviews

A follow-up cycle review verifies a small, targeted fix to a pre-existing issue surfaced during the main run, after the main task has already merged into `main`. The orchestrator may run several such cycles in sequence. Each cycle is independent **in scope** — your prior findings on the main task or an earlier follow-up are not pending items against the new diff, and the verdict restarts with each cycle — but the run-level context you accumulated (the project sources you already read, the main-task work, the circumstances in which the issue surfaced) is exactly why the orchestrator keeps you alive: bring it to bear instead of re-deriving it.

What changes relative to the main-task review:

- **Inputs differ.** The orchestrator's message gives you: a one-line description of the pre-existing issue being fixed, the **cycle's worktree absolute path** at `.claude/worktrees/<cycle-timestamp>/` (distinct from the main run's worktree), the follow-up branch name (`claude/followup/<kebab-slug>`), the iteration number scoped to this cycle (cycles restart at 1), and the filename of the *follow-up orchestration log* at `<cycle-worktree>/orchestration-log/<cycle-timestamp>.md`. There is **no `TASK-*` ID** — the fix targets a pre-existing issue, not a row in `3-code/tasks.md`.
- **Base is `main` post-main-merge.** The orchestrator's message carries the **cycle's base SHA** — the tip of `origin/main` right after the main task's PR merged, the commit the cycle's worktree was created from. That base already contains the just-shipped main-task work; never use a stale pre-merge snapshot, never compare against the moving `origin/main` ref. As on the main task, the developer's fix sits uncommitted: `HEAD` still equals the cycle's base SHA, the inventory comes from `git -C <cycle-worktree> status --porcelain -uall`, tracked modifications from `git -C <cycle-worktree> diff <cycle-base-SHA>`.
- **Skill is `/SDLC-fix`, not `/SDLC-execute-task`.** The governing skill for the cycle is `.claude/skills/SDLC-fix/SKILL.md` — re-read it and hold the cycle to its discipline exactly as you held the main task to `/SDLC-execute-task`'s. `/SDLC-fix` mandates its own implementation log: `<cycle-worktree>/3-code/implementation-log/FIX-<kebab-slug>.md`, started from `_fix_template.md` (not the task template), with the same `## Understanding` + append-only `## Execution log` + final `[CONCLUSION]` rules. Both implementation-log sub-checks (4.(a) faithfulness, 4.(b) rule compliance) apply unchanged to this log; a missing or malformed `FIX-*` log is a finding, exactly as on the main task. The follow-up orchestration log (filename in the orchestrator's message) is the orchestrator's own trail for the cycle — read it for context; it complements, never replaces, the developer's implementation log.
- **Checks that do not apply.** Skip checks specific to the `/SDLC-execute-task` discipline that follow-up fixes cannot satisfy by design: no `tasks.md` row status update, no `Updated` date refresh on a `TASK-*` row.
- **Same keep-alive across iterations** within a cycle — the orchestrator re-notifies you rather than spawning a fresh reviewer, and enforces the iteration cap itself; you simply review whatever you are given. Same verdict vocabulary: approve / changes-requested / blocked.
- **Pre-existing issues you spot during the follow-up review**: call them out separately, as on the main task. The orchestrator does not chain another follow-up to address them — they end up in the termination issue for the user to action.
