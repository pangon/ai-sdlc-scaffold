# Success termination — the `[Orchestrator complete]` issue

Opened only when the run reaches the end of Finalize: commit made, branch pushed, PR opened, PR merged into `main`.

The body is the **only** record of post-`[CONCLUSION]` events. Be specific in every section; an empty section is acceptable when truly nothing happened, but never omit a section.

- **Title**: `[Orchestrator complete] <TASK-id> — <one-line scope summary>`.
- **Label**: `orchestrator-completed`. (Create it via `gh label create` if missing.)
- **Body** (in this order):
  - **Outcome**: PR merged into `main`. Include the merge commit SHA (`git rev-parse origin/main` after merge) and the merged PR number.
  - **Artifacts**: links to `3-code/implementation-log/<TASK-id>.md` and `orchestration-log/<run-datetime>.md` on `main`.
  - **Review iterations**: count from the Decision log.
  - **Development environment installations**: one bullet per `[ENV-INSTALL]` entry the orchestrator appended during the run, in chronological order. For each: tool + version installed, the **chosen pattern (system-wide vs per-project version-pinned)**, the **`DEC-*` filename(s) pinning the pattern** (whether opened by this run or pre-existing), the install command(s) directed, the host OS/distribution, the post-install verification command and its output, and a one-line note on *why the task needed it*. "None" if no installs were authorized.
  - **Post-finalize operations**: a chronological narrative of Finalize steps 2–5 with concrete identifiers — task commit SHA on the working branch, push outcome (remote branch, tip SHA), PR URL, required-checks outcome (passed / none reported, plus total wait time if checks were watched), merge command actually used (`--merge` / `--squash` / `--rebase`), merge result, merge commit SHA on `main`. One short paragraph or tight bullet list.
  - **Follow-up fixes**: one bullet per post-finalize follow-up cycle attempted, in the order they ran. For each: a one-line summary of the pre-existing issue addressed, status (`merged` / `aborted: <reason>`), PR URL, merge commit SHA on `main` (for merged) or branch name (for aborted, plus partial PR URL if a PR was opened before the abort), follow-up review iteration count, the follow-up orchestration log filename, and the cycle's worktree state (`removed` or `left at .claude/worktrees/<cycle-timestamp>/`). "None" if no issues qualified for follow-up or none were attempted.
  - **Team teardown**: outcome of each `SendMessage({type: "shutdown_request"})` (developer + reviewer — `shutdown_response` with timestamp, or "no response" / "teammate already terminated"), `TeamDelete` result, and any orphan resources left in `~/.claude/teams/<team-name>/` or `~/.claude/tasks/<team-name>/` (manual cleanup: `rm -rf ~/.claude/teams/<team-name> ~/.claude/tasks/<team-name>`).
  - **Worktree teardown**: outcome of `git worktree remove <main-worktree-path>` and `git branch -d <main-branch>` (note as anomaly if `-D` was needed), plus the result of `git pull --ff-only origin main` on the main checkout. If worktree removal failed, include the manual cleanup command (`git worktree remove --force <path>` or `rm -rf <path> && git worktree prune`).
  - **Post-`[CONCLUSION]` interruptions**: any `[BLOCKED→RESOLVED]` situation that arose between Finalize step 2's commit and this issue. One bullet per interruption stating what was detected, how it was resolved, and who resolved it; or "None".
  - **Anomalies observed at finalize**: any unrelated drift detected inside the worktree during staging, files explicitly skipped from the commit, worktree-teardown quirks, or any other oddity worth preserving. One bullet per anomaly, or "None".
  - **Pre-existing issues observed (not auto-fixed)**: paraphrase each *non-auto-fixable* pre-existing entry surfaced by the developer or reviewer, with file references — one bullet per issue, or "None". Auto-fixable issues addressed via follow-up cycles appear in the `Follow-up fixes` section above. Issues that qualified for follow-up but were skipped due to the per-run cap are listed here with a `(follow-up deferred — per-run cap)` annotation. Do not open separate `[Pre-existing]` tickets.
  - **Transcript archive**: one bullet per transcript uploaded to `s3://ai-agent-taces/` at Finalize step 9, in upload order — agent name, source JSONL path, destination key (`<repository-name>/<orchestration-log-name>/<agent-name>.jsonl`), outcome. The orchestrator's own transcript is uploaded last and therefore cannot contain this issue's creation. If any upload failed or the step was skipped (aws CLI missing, no credentials, access denied), state that with the error output — the failure is non-fatal and never blocks this issue.
  - **Run metadata**: orchestration-log filename, branch name, worktree path (whether or not still on disk), base SHA at run start, completion timestamp (UTC).

If the issue creation itself fails (e.g., `gh` not authenticated, network error), there is nothing else to fall back on — emit the issue body to stdout as a last-resort signal and terminate.
