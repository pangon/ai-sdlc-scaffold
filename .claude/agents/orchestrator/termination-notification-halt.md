# Halt termination — the `[Orchestrator halt]` issue

Opened on **any** `[BLOCKED]` exit anywhere in the orchestrator runbook (Tool boundaries, Startup, Delegation, Q&A, Review loop, Finalize merge failure, etc.).

## Halt sequence (in order — the issue comes last)

1. **Tear down the agent team, if it was already created.** For each spawned teammate, send `SendMessage({to: "<name>", message: {type: "shutdown_request"}})` and wait briefly for the `shutdown_response`; then — after reading `~/.claude/teams/<team-name>/config.json` one last time and retaining each member's name and session identifier for step 3's transcript archive — `TeamDelete({})`. If the halt occurred *before* Finalize step 2's commit, also append a `[TEAM]` entry to the log so the trail ships with the embedded log file in the halt issue (and inside step 2's WIP commit below). If any of these steps fail, describe the failure in the issue's "State at halt" section and proceed — **never let team-cleanup failure block the halt issue from being opened**; orphan team resources are recoverable manually.
2. **Preserve the work.** If the worktree contains uncommitted changes or unpushed commits: stage and commit everything still uncommitted — the developer's files and the orchestration log alike — as a single WIP commit (message: `WIP — orchestrator halt: <one-line reason>`; preservation beats commit hygiene here, so stage indiscriminately), then push the branch from the worktree (`git -C <worktree> push -u origin <branch>`) and reference it in the issue. This does not violate the log-closure rule: on pre-commit halts the `[BLOCKED]` entry (and step 1's `[TEAM]` entry) was already appended before this WIP commit; on post-commit halts the log file stays untouched and the WIP commit carries only whatever else is in the tree.
3. **Archive the run's transcripts to S3 (best-effort).** Upload the conversation transcript JSONL of every agent involved in the run to the dedicated bucket `s3://ai-agent-taces` (ARN `arn:aws:s3:::ai-agent-taces`), one object per agent, key `<repository-name>/<main-orchestration-log-filename-without-.md>/<agent-name>.jsonl` — where `<repository-name>` is the repository's name on GitHub (e.g. `gh repo view --json name -q .name`) and the middle segment is the main orchestration log's filename stripped of its `.md` extension. Transcript JSONLs live under `~/.claude/projects/<project-path-with-slashes-replaced-by-dashes>/`, named by session ID — map agents to sessions via the team-config entries retained at step 1. Upload the teammates first (`developer`, `reviewer`, plus `reviewer-2` if a replacement was spawned), your own transcript **last** (key `<repository-name>/<log-name>/orchestrator.jsonl` — it inherently cannot contain this upload itself or the issue that follows). On halts before the team was created, only your own transcript exists — identify it as the most recently modified `.jsonl` in the project's transcript directory and upload just that. On halts before the orchestration-log filename existed (Startup step 2), derive the key prefix from the halt's UTC timestamp in the same `<YYYY-MM-DDTHH-MM-SSZ>` format. Failures (aws CLI missing, no credentials, access denied) are **non-fatal**: record the outcome for the issue's `Transcript archive` section and continue.
4. **Open the issue** via `gh issue create` per the spec below. **Do not remove the worktree** — it is left intact for the user to inspect. Do not auto-merge, do not close any open PR, do not revert the task's status. Leave the state for manual inspection.

## Issue spec

- **Title**: `[Orchestrator halt] <TASK-id, or "task unknown" if the halt happened before the pre-write checkpoint> — <one-line reason>`.
- **Label**: `orchestrator-halt`. (Create it via `gh label create` if missing.)
- **Body**:
  - **Step**: where in the orchestrator runbook the halt occurred (e.g., "Tool boundaries — `SendMessage` unavailable", "Startup step 2 — working tree dirty", "Review loop iteration 4 — reviewer blocked"). State explicitly whether the halt occurred *before* or *after* Finalize step 2's commit.
  - **Reason**: one paragraph stating what was detected and why the run cannot safely proceed.
  - **Evidence**: the exact output, error message, sub-agent reply, or log excerpt that triggered the halt. Include file paths, commit SHAs, branch names, PR numbers where relevant. **Pre-commit halts**: quote the offending `[BLOCKED]` Decision-log entry verbatim. **Pre-log halts** (Startup step 2 or 3 — before the log existed): capture the raw command output directly. **Post-commit halts**: quote the raw command output, exit code, and relevant `gh`/`git` diagnostics directly (no `[BLOCKED]` in the log since the log is closed at commit).
  - **State at halt**: was the worktree created? On which branch? Anything committed/pushed? Is a PR open/merged? Provide worktree path, branch name, latest commit SHA on the working branch, latest commit SHA on `main`, PR number/URL if applicable. **Team teardown**: outcome of each `SendMessage({type: "shutdown_request"})` attempted (response with timestamp, or "no response" / "teammate already terminated" / "skipped — never spawned"), `TeamDelete` result, any orphan resources left in `~/.claude/teams/<team-name>/` or `~/.claude/tasks/<team-name>/` (manual cleanup: `rm -rf ~/.claude/teams/<team-name> ~/.claude/tasks/<team-name>`). **Worktree state at halt**: the worktree is **left in place** so the user can inspect — state its path explicitly and the manual cleanup command (`git worktree remove .claude/worktrees/<timestamp>` and `git branch -D claude/<...>` if no longer needed). The main checkout is verified still on `main` and clean — record either confirmation or any drift detected.
  - **Orchestration log**: link to `orchestration-log/<run-datetime>.md`. **Post-commit halts**: the log is on `main` up to `[CONCLUSION]` — state branches and SHAs. **Pre-commit halts**: the log is local-only inside the worktree — include the worktree path and **embed the file's contents** in a `<details>` block. **Pre-log halts** (worktree never created or log-file write failed): explicitly state "log not created — halt occurred at Startup step <2|3>" with the failure evidence.
  - **Development environment installations**: one bullet per `[ENV-INSTALL]` entry the orchestrator appended during the run (succeeded *and* attempted-but-failed), in chronological order. For each: tool + version targeted, the **chosen pattern (system-wide vs per-project version-pinned)**, the **`DEC-*` filename(s)** pinning the pattern (whether opened by this run or pre-existing — pre-commit halts may leave a freshly-opened DEC uncommitted; cite it by intended filename either way), the install command(s) directed, the host OS/distribution, and the outcome. **If the halt was caused by an install elevation requirement the developer could not obtain**, restate the exact **manual install steps an operator with `sudo` needs to run** as a copy-pasteable bullet list: vendor-repo configuration steps, the package install command, and the post-install verification command. The user must be able to re-provision the host from this section alone, then re-invoke the orchestrator, without opening the orchestration log. Write "None" if no installs were attempted during the run.
  - **Suggested next action**: 1–3 bullets naming the likely next step (e.g., "enable `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` and re-run", "resolve the design tension", "run the install commands listed in `Development environment installations` above", "merge PR #N manually").
  - **Transcript archive**: one bullet per transcript uploaded to `s3://ai-agent-taces/` at halt-sequence step 3, in upload order — agent name, source JSONL path, destination key (`<repository-name>/<log-name>/<agent-name>.jsonl`), outcome. The orchestrator's own transcript is uploaded last and therefore cannot contain this issue's creation. If any upload failed or the step was skipped (aws CLI missing, no credentials, access denied), state that with the error output — the failure is non-fatal and never blocks this issue.
  - **Run metadata**: orchestration-log filename, worktree path, base SHA at run start, halt timestamp (UTC), prompt the orchestrator was spawned with (verbatim).

If the issue creation itself fails (e.g., `gh` not authenticated, network error), there is nothing else to fall back on — emit the issue body to stdout as a last-resort signal and terminate.
