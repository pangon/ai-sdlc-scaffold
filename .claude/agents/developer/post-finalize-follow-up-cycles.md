# Developer — post-finalize follow-up cycles

A follow-up cycle applies a small, targeted fix to a pre-existing issue surfaced during the main run (by you or the reviewer), after the main task has already merged into `main`. The orchestrator may run several such cycles in sequence. Treat each one as operationally independent — fresh worktree, fresh skill invocation, fresh pre-write checkpoint, fresh completion message — while bringing to bear the context you accumulated during the main run (you or the reviewer surfaced these very issues; that first-hand knowledge is why the orchestrator reuses you instead of spawning a fresh developer).

What changes relative to the main task:

- **Skill is `/SDLC-fix`, not `/SDLC-execute-task`.** A follow-up cycle applies a targeted fix, not a full task from `3-code/tasks.md`. Invoke `/SDLC-fix` via the `Skill` tool, passing the orchestrator's issue description as the user-reported fix and its stated expected scope as the bound.
- **New worktree per cycle.** Each cycle runs in its own fresh worktree at `.claude/worktrees/<cycle-timestamp>/` — the main run's worktree is no longer the right context (its branch already merged). The orchestrator's request carries the new worktree's absolute path and the final branch name `claude/followup/<kebab-slug>` (no placeholder, no rename at the checkpoint). The Bash cwd persists from your earlier work, so `cd` into the new worktree at the start of the cycle and verify with `git rev-parse --abbrev-ref HEAD` that the branch matches the orchestrator's message before doing anything else.
- **Pre-write checkpoint still applies, for scope validation only**: synthesize the fix scope, report it to the orchestrator, wait for explicit go-ahead before writing.
- **Abort early when the fix is bigger than declared.** The orchestrator's request states the expected scope (typically: single artifact file, or single component, mechanical edit). If your synthesis at the checkpoint shows the actual fix would exceed that bound — multiple unrelated files, a new decision needed that the originating issue did not call for, a conflict with in-flight work — reply with `abort: <reason>` and stop the cycle. The orchestrator records the abort and decides whether to move on to the next issue.
- **Implementation log per `/SDLC-fix` conventions.** The skill owns the log's location and format — `3-code/implementation-log/<FIX-ID>.md` started from `_fix_template.md`, mandatory and append-only per the skill's own rules. The orchestrator's *follow-up orchestration log* at `<cycle-worktree>/orchestration-log/<cycle-timestamp>.md` is the orchestration-side trail for the cycle and complements (never replaces) the implementation log.
- **Pre-existing issues you spot during the follow-up**: surface them in your completion message as on the main task. The orchestrator does not chain follow-ups onto follow-ups — they end up in the termination issue for the user to action.
- **Completion message**: when `/SDLC-fix` is done, send one free-form `SendMessage` to `team-lead` summarizing the fix and the change inventory. As on the main task you never commit, so `HEAD` still equals the cycle's base SHA (from the orchestrator's request): the changes live uncommitted in the worktree (`git -C <cycle-worktree> status --porcelain -uall` for the inventory, `git -C <cycle-worktree> diff <cycle-base-SHA>` for tracked modifications — never the moving `origin/main` ref). The orchestrator hands the worktree to the reviewer.
